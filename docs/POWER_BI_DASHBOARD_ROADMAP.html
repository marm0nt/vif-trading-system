1.2: Understand Your Schema

A star schema organizes data for dashboards:

Think of it like this:
  • Center (Fact): Events that happened (API calls, agent runs, errors)
  • Around (Dimensions): Categories that describe events (which agent? which severity? what time?)

Simplified Tables:

Table Name Purpose Example Data
components What agents, skills, scripts exist watchlist_watcher, api_call, swing_screener
event_types What type of event agent_start, agent_end, api_call, error
severity How serious INFO, WARNING, ERROR, CRITICAL
fact_events ← Center The actual events Timestamp, component, event_type, severity, duration, error_msg
fact_metrics ← Center Measurements (tokens, latency) Timestamp, component, metric_name, metric_value

🛠️ Phase 2: Build the ETL (Week 2)

2.1: What is ETL?

ETL = Extract, Transform, Load

Step What It Does Our Case
Extract Read data from source Read telemetry.jsonl
Transform Clean and organize Parse JSON, normalize to schema
Load Put into database Insert into fact_events table

2.2: The ETL Script

You'll create a Python script that does this automatically. The script is provided in the Markdown version of this roadmap (POWER_BI_DASHBOARD_ROADMAP.md).

File to Create: scripts/telemetry_to_database.py

This script:

📊 Phase 3: Create SQL Queries (Week 2-3)

3.1: What Queries Do

SQL queries ask questions of your database:

  • "How many errors in the last 7 days?"
  • "What's the total cost by agent?"
  • "Which agent is slowest?"

Power BI uses these queries to fill your dashboards with numbers and charts.

3.2: 5 Key Queries

See the Markdown version (POWER_BI_DASHBOARD_ROADMAP.md) for the actual SQL code. Here's what each does:

Query Name Purpose Used In Dashboard
System Health Summary Total events, error count, last update time System Health
Tokens by Agent How many tokens each agent used + cost Cost Analysis
Agent Performance Run count, avg duration, error rate Performance
Errors by Component Error count per component per day Error Tracking
Daily Cost Trend Cost over time (shows trending up/down) Cost Analysis

🔄 Phase 5: Automate (Week 4)

5.1: Schedule ETL to Run Daily

Instead of manually running the ETL script, automate it to run every morning.

For Windows (Task Scheduler):

  1. Create batch file: scripts/run_etl.bat
  2. Open Task Scheduler
  3. Create Basic Task → Daily at 6:00 AM
  4. Action: Run run_etl.bat
  5. Click OK

For Mac/Linux (Cron):

# Edit crontab crontab -e # Add this line (runs daily at 6 AM) 0 6 * * * cd /path/to/vif-trading-system && python scripts/telemetry_to_database.py >> logs/etl_runs.log 2>&1

5.2: Power BI Auto-Refresh

Make Power BI automatically pull fresh data from the database each morning (after ETL completes).

  • Local: Manually refresh, or use Power BI Service with scheduled refresh
  • Cloud: Publish to Power BI Service → Schedule refresh at 6:30 AM

⚠️ Common Pitfalls (Avoid These!)

❌ Overcomplicating the schema

Solution: Start with 3 dimensions + 1 fact. Add complexity only if needed.

❌ Trying SQL Server first

Solution: Start with SQLite (simpler). Upgrade to SQL Server once working.

❌ Loading ALL historical data

Solution: Load last 30 days first. Backfill later if needed.

❌ Building too many dashboards

Solution: Start with 2-3. Quality beats quantity.

❌ Skipping documentation

Solution: Write README. Shows professionalism in interviews.

❌ Not testing the ETL

Solution: Run it manually first. Check logs for errors.
